## 4. 当前最需要完善的部分

下面这些不是“有没有功能”的问题，而是“现在最容易拖后腿的地方”。

### 4.1 P0：配置装载与启动一致性仍有边界问题

这是当前最需要优先修正的一类问题。

#### 现状

- `backend/app/core/settings.py` 仍使用相对路径 `env_file=".env"`。
- 这意味着：
  - 从仓库根目录通过 `start-dev.ps1` 启动时，脚本会先把根目录 `.env` 注入进进程，所以通常没问题。
  - 但如果用户按 `README.md` 里的“单独运行后端 API”方式，从 `backend` 目录直接执行 `uv run fastapi dev main.py`，根目录 `.env` 并不会被自动读取。

#### 影响

- 本地直接单独启动 API / Worker 时，可能出现：
  - Redis 配置为空
  - SMTP 配置为空
  - 其他依赖配置与 `start-dev.ps1` 启动表现不一致

这属于典型的“同一项目，不同启动入口行为不一致”问题。

#### 建议

1. 把 `Settings` 的 `.env` 查找改成明确的项目根路径，而不是依赖当前工作目录。
2. 对 API / Worker / 单脚本入口统一环境装载方式，避免只能靠 `start-dev.ps1` 正常运行。

### 4.2 P0：启动前检查还不够“真实”

#### 现状

- `start-dev.ps1` 目前会检查必填环境变量是否存在。
- 但它不会在启动前主动验证：
  - PostgreSQL 是否真的能连通
  - Redis 是否真的认证通过
  - SMTP 是否真的配置可用

#### 影响

- 当前更像“配置格式检查”，不是“启动前依赖可用性检查”。
- 真实依赖错误会延迟到业务请求阶段才暴露，例如：
  - 注册验证码接口发起时才发现 Redis 认证失败
  - 发邮件时才发现 SMTP 配置不对

#### 现有优点

- 好消息是 `/api/health/readiness` 已经具备 PostgreSQL / Redis / SMTP 探针能力。

#### 建议

1. 把 `start-dev.ps1 -DryRun` 升级为可选的“依赖连通性预检”。
2. 或者在启动成功提示里明确附带一条 readiness 检查建议。
3. 把常见依赖错误前置成更可读的失败信息，而不是等业务请求报栈。

### 4.3 P0：多进程统一执行 schema ensure，开发体验仍偏粗糙

#### 现状

- API、Analysis Worker、Watchlist Worker 三个 Python 入口都会调用 `ensure_database_schema()`。
- 这符合当前运行拓扑文档的设计，但会带来：
  - 启动时重复 schema 检查
  - Alembic 日志重复输出
  - 多进程同时启动时的观感较噪

#### 影响

- 这不一定是功能错误，但会削弱本地启动体验。
- 当数据库较慢或网络较差时，也更容易让用户误判为“程序卡在初始化”。

#### 建议

1. 保留开发模式的便利性，但考虑把启动前迁移和进程启动分离成更清晰的两阶段。
2. 至少减少重复日志或增加更明确的启动进度提示。

### 4.4 P1：后台任务中心已可看，但还不可运维闭环

#### 现状

- 当前任务中心已经支持：
  - 汇总
  - 分页筛选
  - 详情查看
- 但运行拓扑文档也明确说明，第一版暂不支持：
  - 人工取消
  - 人工重试
  - 手工补偿

#### 判断

这说明任务中心现在更像“任务观察台”，还不是“任务运维台”。

#### 建议

1. 下一步优先做失败任务重试。
2. 再补取消 / 重跑 / 补偿入口。
3. 对关键任务增加更细的状态机与执行日志切片，而不只是汇总字段。

### 4.5 P1：分析模块已经有运行元数据，但还缺质量治理闭环

#### 现状

- 已经记录了：
  - `prompt_version`
  - `model_name`
  - `reasoning_effort`
  - token usage
  - cost estimate
  - failure type

这已经比“黑盒分析”前进了一大步。

#### 仍然缺少

- 正式的评测样本集
- Prompt / 模型回归对比机制
- 分析质量验收标准
- 失败类型与结果质量的持续跟踪面板

#### 判断

分析模块已经值得继续深挖，但接下来不应只堆能力，而要把“质量治理”做起来，否则复杂度会越来越高、结果可控性却跟不上。

### 4.6 P1：研究域扩展开始成型，但还没有形成更强的研究资产层

#### 已完成

- 市场主题域模型
- 个股主题接口
- 热点候选股主题证据增强

#### 仍然不足

- 概念 / 题材 / 成分股层还不够深
- 研究证据的来源稳定性、可信度分层还可以继续做
- 个股详情页距离“研究主页”还有一段距离

#### 建议

1. 按 `PROGRESS.md` 中的建议，优先评估题材/概念成员增强。
2. 把主题、概念、候选证据与个股研究页更深地串起来。

### 4.7 P2：结果交付能力已起步，但还没形成“成果沉淀产品”

#### 已完成

#### 判断

当前更像“工作台内完成研究”，还不是“研究成果可以顺畅传播和沉淀”的状态。

### 4.8 P2：工程化文档已改善，但交付材料仍以开发者视角为主

当前已经有：

## 5. 当前阶段的优先级建议

如果按“最值得投入”的顺序来排，我建议这样推进：

- Prompt / 模型回归样本
- 分析质量指标

- 题材 / 概念 / 成分股增强
- PDF / 分享 / 成果沉淀
- 更完整的研究资产层

## 6. 一句话总结

`StockProject` 当前已经不是“做完几个页面和几个接口”的阶段，而是已经形成了：

- **完整的股票研究产品雏形**
- **比较扎实的后端分层与测试基线**
- **明显加强过的工程化底座**

接下来最关键的，不是盲目再加新功能，而是：

1. 先把 **配置装载、启动一致性、依赖预检** 做稳  
2. 再把 **任务中心与分析质量治理** 做深  
3. 最后把 **研究资产扩展和结果交付** 做强  

如果按这个顺序推进，项目会更像一个持续可维护、可放大的产品，而不仅仅是功能丰富的开发仓库。
