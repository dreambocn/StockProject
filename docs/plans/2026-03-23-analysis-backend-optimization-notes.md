# 分析链路后端优化建议（P1 / P2）

## 背景

当前 `GET /api/analysis/stocks/{ts_code}/summary` 已能稳定支撑 P0 分析工作台，但从服务边界、查询副作用和长期数据治理角度看，后端仍有继续优化空间。

本文件只记录后续建议，不包含本轮实现。

## P1：应优先处理

### 1. 将 `summary` 接口改为纯读接口

- 当前 `analysis_service.py` 在读取 `summary` 时会同时完成：
  - 事件情绪增强
  - 价格窗口关联
  - 因子权重计算
  - LLM 摘要生成
  - `analysis_event_links` / `analysis_reports` 落库
- 这会让查询接口带有写副作用，不利于缓存、幂等控制和排障。
- 建议拆分为：
  - **命令链路**：刷新 / 物化分析结果
  - **查询链路**：只读取已落库结果并返回 summary

### 2. 为分析报告增加去重或更新策略

- 当前多次触发分析可能持续新增 `analysis_reports` 记录。
- 长期运行后会出现：
  - 同一股票同一主题重复报告堆积
  - 前端难以判断“最新有效版本”
  - 存储与查询成本上升
- 建议按 `ts_code + topic + 时间窗` 做 upsert 或版本化保留策略。

### 3. 事件锚点改为真实交易日

- 当前价格窗口起点使用 `published_at.date()`，在节假日、周末、盘后公告场景会产生偏差。
- 建议基于交易日历，将锚点统一改为“事件发布时间后的首个有效交易日”。

## P2：建议继续收敛

### 4. 对前端补充稳定展示元信息

- 后续建议补充以下只读字段，减少前端推导：
  - `report_mode` / `is_fallback`
  - `source_url`
  - `top_factor_key`
  - `sorted_rank`
  - `staleness`

### 5. 继续拆分分析服务边界

- 当前 `analysis_service.py` 同时承担：
  - 查询聚合
  - 事件增强
  - 关联计算
  - 报告生成
  - 持久化写回
- 建议按职责拆成独立服务，降低单文件复杂度，提升后续测试稳定性。

### 6. 增加缓存与幂等控制

- 对相同 `ts_code + query` 的短时间重复请求补缓存或幂等控制。
- 重点避免刷新按钮频繁触发重复 LLM 调用和重复写库。

## 推荐推进顺序

1. 先拆 `summary` 读写职责
2. 再补报告去重 / upsert
3. 然后修正交易日锚点
4. 最后补展示元信息、缓存与幂等
